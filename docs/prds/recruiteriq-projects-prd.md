# RecruiterIQ Projects — Product Requirements Document (PRD)

## Requirements Description

### Background

- **Business Problem**: Recruiters use RecruiterIQ tools in isolation — there is no way to organise work around a specific role, share progress with teammates, track a candidate pipeline across scoring/assessments, or review prior searches. Every session starts from scratch. Projects fix all of this by making the role opening the organising unit, and tools become utilities called from within a project.
- **Target Users**:
  - **Recruiters** (Agency plan): create projects, add candidates, run scoring and red flag checks, generate Boolean variations, send assessments
  - **Managers / Owners** (Agency plan): all recruiter actions plus see all Boolean variations for a project, regenerate all variations, share projects, access team member management
  - **Viewers** (any plan, shared access): read-only access to a project they've been granted access to
- **Value Proposition**: One project page replaces disconnected tool sessions. Candidates are ranked automatically, Boolean strings don't overlap between teammates, and managers get a live view of every role in the pipeline without asking for status updates.

---

### Feature Overview

- **Core features**:
  1. My Projects page with filter/search/sort
  2. Create Project form
  3. Project Detail page — 5 tabs: Candidates, Job Description, Boolean Strings, Activity Feed, Settings
  4. Boolean Variation System — unique per-recruiter search strings generated by Claude
  5. Candidate pipeline — add, auto-score, manual red flag check, send assessment
  6. Batch scoring — score all unscored candidates once JD exists
  7. Project sharing with role-based access (Collaborator / Viewer)
  8. In-app + email notifications on project share
  9. Activity feed logged automatically on every write action
  10. Assessment builder overlay — create assessment without leaving project

- **Feature Boundaries**:
  - Projects are the primary workflow — existing tool pages remain available as standalone utilities
  - No ATS or Bullhorn integration — RecruiterIQ is fully standalone
  - Assessment builder opened from a project renders in a full-screen overlay; after publish, control returns to the project
  - Batch scoring fires sequentially with live progress; background continuation if recruiter navigates away
  - Red flag check is manual per candidate (not automatic on add) to conserve AI credits
  - Boolean variations are per-recruiter, not per-project — each recruiter sees only their own active variation
  - History page "Projects" tab mirrors the My Projects page card grid filtered to the viewing user

- **Out of scope**:
  - Candidate messaging / email outreach from within a project
  - Importing candidates in bulk via CSV (post-MVP)
  - Candidate-facing project portal
  - Connecting a project to an external job board posting

#### Tab 1 — Candidates

**Add Candidate button** → opens slide-over panel (right side, 480px wide):

| Field | Type | Rules |
|-------|------|-------|
| Candidate Name | text input | Required |
| Candidate Email | text input | Required, valid email format |
| Resume | FileDropTextarea | Required, PDF/DOCX/TXT, max 5000 words |

On save:
- Unique constraint check: if `(project_id, candidate_email)` already exists → hard block with error: *"This candidate is already in this project."* + link that scrolls to their row
- If project **has JD**: call `/api/score-resume` with `jd_text` + `resume_text`, save `cqi_score` + `cqi_breakdown_json` to `project_candidates`
- If project **has no JD**: show warning banner above table: *"Add a job description to enable auto-scoring. You can still add candidates now."* Candidate is added with `cqi_score = null`
- Log to `project_activity`: `"[User] added [Candidate Name]"` (if scored: `"(CQI: 84)"`)
- Slide-over closes, new candidate appears at correct rank position in table

**"Score All Unscored Candidates" button** (appears when project gains a JD and unscored candidates exist):
1. Confirmation modal: *"This will score N candidates using N AI credits. Continue?"* — Cancel | Score All (N credits)
2. After confirming: progress bar above table — *"Scoring candidates… 3 of 8 complete"* — animated indigo bar
3. Each candidate row updates in real time as their score returns
4. On full completion: toast *"All candidates scored successfully"* + progress bar disappears
5. On partial failure: skip failed, continue rest, warning at end: *"7 of 8 scored — 1 failed, click to retry"*
6. If recruiter navigates away: scoring continues in background (fire-and-forget after confirmation); live progress only visible if they stay on page

**Candidates table columns**:

| Column | Notes |
|--------|-------|
| Rank | Auto-assigned by CQI score descending; `—` if unscored |
| Name | Clickable → opens View Resume slide-over |
| Email | Plain text |
| CQI Score | Score ring (small) + numeric value; `—` if unscored |
| Red Flags | If run: severity dot + count (e.g. 🔴 3 flags); if not run: `—` with subtle "Check" link |
| Assessment | Not Sent / Pending / Completed (Trust: XX · Skill: XX) |
| Added By | User avatar + name |
| Actions | Three-dot menu: View Resume · Check Red Flags · Generate Summary · Send Assessment · Remove |

**Actions per candidate**:

- **View Resume**: opens right slide-over with full resume text + CQI score breakdown (category bars)
- **Check Red Flags**: calls `/api/analyze-red-flags` with `resume_text` + project `jd_text` (optional); saves `red_flag_score` + `red_flag_summary` to `project_candidates`; updates Red Flags column inline; logs to activity
- **Generate Summary**: calls `/api/generate-summary` pre-filled with candidate resume + project JD as notes; output shown in a modal with copy button; does not save to `client_summaries` unless recruiter explicitly clicks "Save to History"
- **Send Assessment**: opens assessment selector modal:
  - Lists all published assessments owned by the team
  - "Create new assessment" option → opens assessment builder in full-screen overlay (`fixed inset-0 z-50` over the project page); after publish, overlay closes, new assessment pre-selected in selector
  - After selecting: sends invite via existing `/api/assessments/invite` route; `assessment_invite_id` saved to `project_candidates`; logs to activity: *"[User] sent assessment to [Candidate Name]"*
- **Remove**: confirmation dialog → soft-delete candidate from `project_candidates` (set `deleted_at`); logs to activity

**Empty state**: "No candidates yet — add your first one" + Add Candidate button

### Boolean Variation System

**API Route: `/api/projects/generate-variations` (POST)**

Request body:
```json
{
  "project_id": "uuid",
  "jd_text": "string",
  "job_title": "string",
  "recruiter_ids": ["uuid", "uuid", ...]
}
```

Behaviour:
- `recruiter_ids` drives how many variations to generate (min 1, no upper cap)
- If generating a **full set**: Claude generates one variation per recruiter_id, each using different combinations of:
  - Job title synonyms (e.g. "React Developer" / "Frontend Engineer" / "UI Engineer")
  - Skill permutations (React + TypeScript / ReactJS + JavaScript / React.js + Node.js)
  - Seniority expressions (Senior / "5+ years" / Lead / Principal)
  - Exclusion term variations (NOT junior / NOT entry-level / NOT intern)
- If generating **for a new recruiter added to existing project** (11th+): generate 1–2 new variations, passing existing active strings to Claude as context to ensure no overlap; save and assign immediately
- Each variation: `linkedin_string` + `indeed_string`
- Set all new variations to `is_active = true`, all prior active rows for those user_ids to `is_active = false`
- Return the variation assigned to the requesting user

**Trigger points**:
- Project creation with JD provided → auto-triggered after redirect (background, non-blocking)
- Manual "Generate Boolean Variations" CTA on Boolean tab (when JD exists, no active variations)
- New recruiter added to project (if active variations already exist for others) → generates 1 new variation for them only
- "Regenerate My String" → generates 1 new variation for requesting user
- "Regenerate All" → generates full new set for all current project members

CREATE TABLE public.project_candidates (
  id                    uuid        DEFAULT gen_random_uuid() PRIMARY KEY,
  project_id            uuid        REFERENCES public.projects(id) ON DELETE CASCADE NOT NULL,

  candidate_email       text        NOT NULL,

#### RLS Policies (all 5 tables)

Users can read/write projects they own or are a member of:

```sql
-- projects: visible if owner or project_member
CREATE POLICY "projects_select" ON public.projects FOR SELECT
  USING (
    owner_id = auth.uid() OR
    EXISTS (SELECT 1 FROM public.project_members
            WHERE project_id = id AND user_id = auth.uid())
  );

CREATE POLICY "projects_insert" ON public.projects FOR INSERT
  WITH CHECK (owner_id = auth.uid());

CREATE POLICY "projects_update" ON public.projects FOR UPDATE
  USING (
    owner_id = auth.uid() OR
    EXISTS (SELECT 1 FROM public.project_members
            WHERE project_id = id AND user_id = auth.uid()
            AND role IN ('owner','collaborator'))
  );

CREATE POLICY "projects_delete" ON public.projects FOR DELETE
  USING (owner_id = auth.uid());

-- project_candidates: same membership check
CREATE POLICY "project_candidates_select" ON public.project_candidates FOR SELECT
  USING (
    EXISTS (SELECT 1 FROM public.projects p
            LEFT JOIN public.project_members pm ON pm.project_id = p.id
            WHERE p.id = project_id
            AND (p.owner_id = auth.uid() OR pm.user_id = auth.uid()))
  );

-- (Insert/update/delete policies follow same pattern, restricted to collaborator+ role)
```

## Design Decisions

### Technical Approach

- **Architecture**: Projects layer sits on top of all existing tool routes. Existing `/api/score-resume`, `/api/analyze-red-flags`, `/api/generate-summary`, `/api/assessments/invite` are called as-is; projects API routes orchestrate them and persist results to `project_candidates`
- **Assessment builder overlay**: The existing `<AssessmentBuilder>` component is rendered inside a `fixed inset-0 z-50` full-screen container mounted on the project detail page. After publish event, the overlay unmounts and the parent re-fetches published assessments for the selector
- **Boolean variation generation**: New Claude prompt specifically designed for variation diversity — passes existing active strings as context when generating incremental additions to prevent overlap
- **Batch scoring**: Sequential (not parallel) to avoid rate-limit issues on the Anthropic API. Each call awaits the previous before starting next

### Constraints

- **Performance**: Project detail page candidate table should handle up to 50 candidates without pagination for MVP; add pagination if >50 becomes common
- **Boolean tab manager view**: If a project has 20+ recruiters, the variations table will be long — add vertical scroll with fixed header
- **is_active enforcement**: Application-level (not DB constraint) — before inserting a new active variation for a user, UPDATE prior rows for that (project_id, user_id) pair to `is_active = false`

### Risk Assessment

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Batch scoring burns credits on large pipelines | Medium | Medium | Confirmation modal shows credit count; users opt in explicitly |
| Boolean variation Claude prompt produces duplicates | Low | Medium | Pass existing strings as context; add dedup check before saving |
| Assessment overlay state loss on unmount | Low | High | Persist builder draft to localStorage keyed by project_id |
| RLS policy gaps on shared projects | Medium | High | Integration test every role combination before shipping |
| Notifications enum migration fails on existing db | Low | High | Use CHECK constraint update pattern, not ALTER TYPE, if needed |
